\section{Guiding Principles}

\begin{itemize}
\item Define syntactic subset of Python called \Lang
\item Define pure semantics for \Lang
\item The semantics of a \Lang program should be the same as its semantics in Python
\end{itemize}

\section{Syntax}

\subsection{Environments}

Borrowing terminology from Java, environments $\Gamma, \Delta$ are (not necessarily finite) maps from sets of
variables to their \emph{definite assignment} status, which is an element of the 2-point Boolean lattice
$\{\statusDefAssigned, \statusNotDefAssigned\}$. The top element $\statusDefAssigned$ indicates
\emph{definitely assigned}, and the bottom element $\statusNotDefAssigned$ indicates \emph{not definitely
assigned}; we can think of these as degenerate types (``any'' and ``none'').

\paragraph{Environment operations.}

Write $\top$ for the environment that maps every variable to $\statusDefAssigned$, and $\Gamma \runion \Delta$
for the right-based map union of $\Gamma$ and $\Delta$. Also define
%
\[\Gamma \merge \Delta := \{x: \bot \mid \forall x \in (\Gamma \setminus \Delta) \cup (\Delta \setminus \Gamma)\} \cup \{x: \Gamma(x) \meet \Delta(x) \mid x \in \Gamma \cap \Delta\}\]

\noindent The $\top$ environment is the left and right unit for $\merge$.

\subsection{Examples}
\subsubsection{Unreachable code}
Currently permitted (but could easily be outlawed):

\begin{python}
def foo(x):
  return x + 1
  return y    # y undefined, but ok since this code is unreachable
\end{python}

\subsubsection{Conditionals without \pyinline{else} clauses}

An \pyinline{if} statement without an \pyinline{else} clause is equivalent to an \pyinline{if}
statement with an \pyinline{else} clause whose body is a $\exPass$ statement.

\subsubsection{Local assignment in conditional clauses}

For flexibility, we would like to permit ``branch-local'' variables, so that the assignment of \pyinline{x} below
(left) is permitted and remains local to that branch of the conditional:

\begin{figure}[H]
\centering
\begin{subfigure}{0.48\textwidth}
\centering
\begin{python}
def foo():
  z = 6
  if b:
    x = 3
    y = x + 1
  else:
    y = 0
  return z
\end{python}
\caption{Branch-local definition of \pyinline{x}}
\end{subfigure}
\hfill
\begin{subfigure}{0.48\textwidth}
\centering
\begin{python}
def foo():
  x = 6
  if b:
    x = 3
    y = x + 1
  else:
    y = 0
  return x # error: partially defined
\end{python}
\caption{Now with use of \pyinline{x} after the conditional}
\end{subfigure}
\end{figure}

\noindent The \pyinline{x} in \pyinline{return x} refers to the outer \pyinline{x}, and so even when \pyinline{b} is
\pyinline{True}, the function would return \pyinline{6}. However, in Python the function would return \pyinline{3}, so the
same program would have different meanings in Python and \Lang.

Options:
\begin{enumerate}
\item Require \pyinline{x} to be defined in each branch of the conditional (quite restrictive);
\item Rule out the use of \pyinline{x} after the conditional, because it is only ``partially shadowed''
\end{enumerate}

\noindent (2) would still support (apparently) branch-local variables, but they are not really: they
\emph{always} shadow variables from the enclosing scope, either partially or completely. Partial shadowing is
permitted as long as the variable is not used after the conditional.
